Overview

wg-easy exposes a small REST API for managing WireGuard peers. All paths are served from the same origin as the dashboard. JSON is used for request/response bodies; binary endpoints (QR, .conf download) return the appropriate Content-Type.

All /api/wireguard/** endpoints require an authenticated session when the server was started with the PASSWORD environment variable. When PASSWORD is unset, the API is open.

Authentication

Auth is cookie-based. POST /api/session with the password sets a connect.sid cookie that subsequent requests must send. From a browser, cookies are managed automatically. From curl, use a cookie jar: 

# Login (saves cookie)
curl -c cookies.txt -X POST http://localhost:51821/api/session \
  -H "Content-Type: application/json" \
  -d '{"password":"YOURPASS"}'

# Use the saved cookie for subsequent calls
curl -b cookies.txt http://localhost:51821/api/wireguard/client

GET /api/release

Returns the numeric release identifier. Public.

Response 200

8

GET /api/openapi.json

Returns the full OpenAPI 3.0 specification for this API. Public.

GET /api/session

Reports session state. Public.

Response 200

{
  "requiresPassword": true,
  "authenticated": false
}

POST /api/session

Login. Sets a connect.sid cookie on success.

Request body

{ "password": "YOURPASS" }
StatusMeaning
204Authenticated.
401Missing or wrong password.

DELETE /api/session

Logout. Destroys the session.

GET /api/wireguard/client

List all clients. Each entry includes live transfer counters and current schedule state from wg show wg0 dump.

Response 200 (excerpt)

[
  {
    "id": "f6c1...",
    "name": "phone",
    "enabled": true,
    "address": "10.8.0.2",
    "publicKey": "abc...=",
    "createdAt": "2026-04-26T10:00:00.000Z",
    "updatedAt": "2026-04-27T08:00:00.000Z",
    "schedule": { "enabled": false, "timezone": "UTC", "days": { "monday": { ... }, ... } },
    "scheduleActive": true,
    "transferRx": 12345,
    "transferTx": 67890,
    "latestHandshakeAt": "2026-04-27T07:55:00.000Z",
    "persistentKeepalive": "off"
  }
]

POST /api/wireguard/client

Create a new client. The server generates the keypair and PSK and allocates the next free IPv4. Defaults: enabled=true, schedule disabled.

Request body

{ "name": "phone" }

Response 200 — full server-side record (includes privateKey and preSharedKey).

DELETE /api/wireguard/client/{clientId}

Permanently delete a client. Idempotent (no error if it already does not exist).

POST /api/wireguard/client/{clientId}/enable

Set enabled=true. Note: the peer is only added to wg0.conf if the schedule (when enabled) currently allows it.

POST /api/wireguard/client/{clientId}/disable

Set enabled=false. Removes the peer from wg0.conf.

PUT /api/wireguard/client/{clientId}/name

Rename a client.

Request body

{ "name": "laptop" }

PUT /api/wireguard/client/{clientId}/address

Change the client IPv4 address inside the WireGuard tunnel.

Request body

{ "address": "10.8.0.42" }

Returns 400 if the address is not a valid IPv4.

GET /api/wireguard/client/{clientId}/qrcode.svg

Returns a 512×512 SVG QR code that encodes the client .conf. Content type: image/svg+xml.

GET /api/wireguard/client/{clientId}/configuration

Download the WireGuard .conf file. Sends Content-Disposition: attachment.

PUT /api/wireguard/client/{clientId}/schedule

Replace the per-client schedule. Missing days are filled with { active: false, start: "00:00", end: "23:59" }. Time strings are normalized to HH:MM; unknown timezones fall back to UTC.

A 60-second background ticker rebuilds wg0.conf and runs wg syncconf, so peers are removed and restored as schedule boundaries pass — no client-side action needed.

Request body

{
  "schedule": {
    "enabled": true,
    "timezone": "Asia/Jakarta",
    "days": {
      "monday":    { "active": true,  "start": "08:00", "end": "17:00" },
      "tuesday":   { "active": true,  "start": "08:00", "end": "17:00" },
      "wednesday": { "active": true,  "start": "08:00", "end": "17:00" },
      "thursday":  { "active": true,  "start": "08:00", "end": "17:00" },
      "friday":    { "active": true,  "start": "08:00", "end": "17:00" },
      "saturday":  { "active": false, "start": "00:00", "end": "23:59" },
      "sunday":    { "active": false, "start": "00:00", "end": "23:59" }
    }
  }
}

Notes

  • If end < start on a day, the active window wraps past midnight (e.g. 22:00 → 02:00).
  • When schedule.enabled is false, the schedule is ignored and the client follows only the master enabled flag.
  • The schedule does not override the master enabled flag — both must be true (and the day must be active) for the peer to appear in wg0.conf.

Schema: Client

FieldTypeDescription
idstring (uuid)Server-assigned client id.
namestringDisplay name.
enabledbooleanManual master enable flag.
addressstring (IPv4)Tunnel IPv4 (e.g. 10.8.0.2).
publicKeystringWireGuard public key.
createdAtstring (ISO date)Creation timestamp.
updatedAtstring (ISO date)Last update timestamp.
latestHandshakeAtstring|nullLast handshake from wg show dump.
transferRxinteger|nullBytes received from peer.
transferTxinteger|nullBytes sent to peer.
persistentKeepalivestring|nullReported keepalive setting.
scheduleScheduleSee below.
scheduleActivebooleanTrue now according to schedule (or no schedule restriction).

Schema: Schedule

FieldTypeDescription
enabledbooleanMaster toggle. When false, the schedule is ignored.
timezonestringIANA timezone (e.g. Asia/Jakarta). Defaults to UTC; unknown zones are normalized to UTC.
daysobjectRequired keys: monday…sunday. Each maps to a ScheduleDay.

ScheduleDay

FieldTypeDescription
activebooleanWhether the client is allowed on this day.
startstring (HH:MM)Start of the active window.
endstring (HH:MM)End of the active window. If end < start, the window wraps past midnight.
